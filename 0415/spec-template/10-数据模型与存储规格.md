## 1. 概念模型

```
Report（研报）1 ── * ReportChunk（文档片段）
   │                      │
   │  report_id (PK)      │  chunk_id (PK)
   │  title               │  report_id (FK → Report)
   │  company_name        │  content / vector
   │  stock_code          │  page / metadata
   │  broker / analyst    └  embedding_vector
   │  rating / target_price
   │  status / file_path
   └  created_at

Session（会话）1 ── * ChatMessage（聊天消息）
   │                      │
   │  session_id (PK)     │  message_id (PK)
   │  title               │  session_id (FK → Session)
   │  report_ids[]        │  role / content
   │  created_at          │  citations[]
   │  updated_at          │  answer_source
   └                      └  created_at

Stock（股票）1 ── * Report（研报）
   │
   │  code (PK)
   │  name / industry
   │  price_data
   │  indicators
   └  updated_at

Annotation（标注）* ── 1 Report（研报）
   │
   │  annotation_id (PK)
   │  report_id (FK)
   │  page / text / note
   └  created_at

Favorite（收藏）* ── 1 Report（研报）
   │
   │  favorite_id (PK)
   │  report_id (FK)
   │  tags[]
   └  created_at
```

> **核心关系**：一个 Session 可包含零到多条 QARecord。外键通过 `session_id` 字段逻辑引用。

## 2. 存储引擎

| 项 | 说明 |
|----|------|
| 引擎 | JSON 文件（`json` 标准库） |
| 编码 | UTF-8，缩进 2 空格 |
| 读写模式 | RMW（全量读入 → 修改 → 全量写回） |
| 并发安全 | 无锁，不支持并发写入 |
| 文件布局 | `{DATA_DIR}/reports.json` + `sessions.json` + `messages.json` + `stocks.json` + `annotations.json` + `favorites.json` + `vectors/`（向量索引） |

## 3. 实体定义 — Session

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `session_id` | `string` (UUID) | 是 | **PK** | 会话唯一标识 |
| `title` | `string` | 是 | ≤23 字符 | 默认“新会话”，首次问答自动重命名 |
| `report_ids` | `array<string>` | 是 | 元素为存在的 report_id | 关联研报列表，对应 `09` 3.1 响应字段 |
| `created_at` | `string` (ISO-8601) | 是 | UTC+Z | 创建时间 |
| `updated_at` | `string` (ISO-8601) | 是 | UTC+Z | 最后更新时间 |
| `query_count` | `integer` | 是 | ≥ 0 | 累计问答次数 |

## 4. 实体定义 — QARecord

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `id` | `string` | 是 | **PK**，`rec_{ts}` | 基于时间戳生成 |
| `session_id` | `string` | 是 | **FK** → Session | 所属会话 |
| `query` | `string` | 是 | 1–500 字符 | 用户提问原文 |
| `answer` | `string` | 是 | — | LLM/Demo 返回的答案 |
| `llm_used` | `boolean` | 是 | RULE-005 | 是否使用真实 LLM；**仅存储层使用，不出现在 `09` API 响应中** |
| `model` | `string\|null` | 是 | — | 模型标识；**仅存储层使用** |
| `response_time_ms` | `integer` | 是 | SLO < 5000 | 响应耗时（ms）；**仅存储层使用** |
| `answer_source` | `string\|null` | 否 | 枚举 | `copaw`\|`bailian`\|`demo`；**仅存储层使用，不出现在 `09` API 响应中** |

> **存储字段 vs API 字段映射说明**：
> `09` 第3章消息接口返回的是 `message_id`、`role`、`content`、`citations`、`created_at`，对应关系如下：
>
> | QARecord 存储字段 | `09` API 响应字段 | 说明 |
> |---|---|---|
> | `id` | `message_id` | 存储层用 `id`，API 层暴露为 `message_id` |
> | `query` / `answer` | `content` | API 层用统一的 `content` 字段，存储层按 `role` 区分写入 `query`（user）或 `answer`（assistant） |
> | `llm_used` | 不暴露 | 存储层内部记录 |
> | `model` | 不暴露 | 存储层内部记录 |
> | `response_time_ms` | 不暴露 | 存储层内部监控 |
> | `answer_source` | 不暴露 | 存储层内部记录 |
> | `citations` | `citations` | 直接映射 |
> | `created_at` | `created_at` | 直接映射 |

## 6. 关键业务逻辑

**示例 — 首次问答自动命名**：

```python
if session["query_count"] == 1:
    session["title"] = query[:20] + ("..." if len(query) > 20 else "")
```

> **规则**：仅首次问答（query_count 从 0 → 1）触发，后续不再自动更名。

| 逻辑 | 触发条件 | 行为 |
|------|----------|------|
| 首次问答自动命名 | 会话首次收到用户消息 | 标题 = 首条消息前 20 字 + `...` |
| 级联删除 | 删除 Session | 同步删除所有关联 ChatMessage |
| 研报删除级联 | 删除 Report | 同步删除关联 Chunks、Annotations、Favorites |
| 解析状态流转 | 上传完成后 | pending → processing → completed/failed |
| 向量索引更新 | 研报解析完成 | 自动构建文档片段向量索引 |
| 股票数据缓存 | 首次查询股票 | 缓存 30 秒，过期后自动刷新 |
| 收藏去重 | 调用 `add_favorite` 前 | 检查同一 `report_id` 是否已存在收藏记录；重复收藏时抛 `DuplicateFavoriteError`，防止 `favorites.json` 出现重复条目 |
| 对比范围校验 | 调用 `POST /compare` 时 | 校验所有 `report_ids` 对应的 `company_name` 必须相同（对应 AC-005-2）；不一致时展层返回 422 + `CROSS_COMPANY_COMPARE` |
| `query_count` 自增 | 每次 `add_message` 且 `role = "user"` | 将对应 session 的 `query_count` + 1，并触发首次问答命名判断 |

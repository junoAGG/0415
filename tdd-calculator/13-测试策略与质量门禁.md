## 一、测试分层

> 共 3 层，写测试前先搞清楚每层验什么。

| 层级 | 工具 | 覆盖对象 | 说明 |
|------|------|----------|------|
| **L1 Unit** | Jest | 单个方法的数学逻辑 | 验证 add/subtract/multiply/divide/power 的计算正确性 |
| **L2 Validation** | Jest | 参数校验与错误处理 | 验证非法输入（string/null/NaN 等）能正确抛出异常 |
| **L3 State** | Jest | 状态管理 | 验证 getResult/clear 及运算后状态一致性 |

## 二、用例拆分原则

> **核心理念：数"可独立失败的点"** — 每条 TC 只回答一个问题，失败时不看代码就知道哪里坏了。

| 来源 | 计数方法 | 示例 |
|------|----------|------|
| 正常路径 | 每个运算的 happy path = 1 条 | `add(2,3) → 5` = TC-001 |
| 符号组合 | 正正/正负/负负/零 每组合 = 1 条 | `add(-2,-3) → -5` = TC-003 |
| 错误码 | 每个 distinct error message = 1 条 | `string → '参数必须是数字'` = TC-031 |
| 边界值 | 每个特殊数值 = 1 条 | `Infinity, MAX_SAFE_INTEGER` = TC-037 |
| 业务异常 | 每个业务规则异常 = 1 条 | `divide(10,0) → '除数不能为零'` = TC-025 |

## 五、质量门禁

| Gate-ID | 条件 | 阻塞合并 |
|---------|------|----------|
| **G-LINT** | ESLint 通过，0 errors, 0 warnings | **是** |
| **G-UNIT** | Unit 测试全绿（排除 Red 阶段文件），覆盖率 ≥ 80% | **是** |
| **G-COV** | Statements 100%, Branches 100%, Functions 100%, Lines 100% | **是** |
| **G-TDD** | 每个新功能必须先有 Red 阶段的失败测试，再有 Green 阶段的通过实现 | **是** |
| **G-PERF** | 全部测试执行时间 < 10s | 否（warning） |

## 六、TDD 适用场景

| 场景 | TDD 价值 | 建议 |
|------|----------|------|
| 基础四则运算 | 中 | 建议 TDD，快速验证数学正确性 |
| 参数校验 / 错误处理 | 中 | 建议 TDD，先定义异常边界 |
| 幂运算 / 复杂数学 | **高** | **强烈建议 TDD**，边界条件多 |
| 金融计算 / 精度要求 | **极高** | **必须 TDD**，浮点精度必须逐一验证 |
| 状态管理 | 低 | 写完补测试即可 |
